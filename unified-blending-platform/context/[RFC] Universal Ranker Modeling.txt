What
We propose a single ranker (e.g. universal) to jointly rank carousels, stores, collections and deal carousels
Why?
Multiple experiments and analyses evaluating the diversity and ranking efficiency of core discovery surfaces suggest that the current system of fixing carousels above home feed and using a single ranker is suboptimal. In particular we see that stores with low probability of conversion (pConv) have high visibility while higher pConv stores in home feed require a lot of scrolling to find. 


Today the carousel ranker ranks carousels by calculating a probability of conversion for each carousel and then ranking the carousels, however carousels are always ensured to be placed above the home feed. In particular conversions for carousels are treated independently and require calibrated probabilities.


From past experiments, we believe that the flexibility and holistic objective of a joint “universal” ranker will lead to a ~0.1% absolute lift in conversion. Additionally, cross-vertical initiatives (e.g. new homepage) will later benefit from this architecture by being able to rank disparate types of entities together.


  

Goals
Define an entity [c][d]as anything that we can place in a vertical position on the home feed. For example a single store[e][f], a store carousel (single entity because it takes up 1 vertical slot), or a collections tile. We then propose a ranker that:


* Jointly ranks stores, carousels, and collections to ensure the ones that have the highest pConv[1] are ranked at the top.
* Easily generalizes to new types of entities as long as they can be expressed as a sequence of embeddings.
* Supports ranking overrides (for example to boost a particular entity type), providing an estimated loss for each.
* Does not increase p95 latency of homepage by more than 10%.
* Achieves incremental gains in global conversion and/or order rate, alongside homepage conversion improvements.

Non-goals
The following are out of scope of this proposal:


* Improving deduplication. To avoid having too many moving pieces, we will reuse the current implementation (described here).
* Immediately handle impression fatigue. While including features such as view counts can help provide some impression discounting, this is not an explicit goal of this model. And related measures to improve the overall recommendation experience are better handled through a post-porcessing hoisting procedure which will allow for more sophisticated de-duplication and recommendation diversity
* This architecture is flexible to handle incentive information for ads and promotions (see the general implementation discussion below) but is out of scope for our MVP
Design[m][n]
Architecture
Conceptually, the overall ranker’s architecture consists of two sequential stages:


1. Scoring, where entities are scored by an ML model;
2. Hoisting, where diversification rules are combined with the above scores to produce the final ordering of entities.
  



Multiple-step ranking systems are common in the search literature (Ebay, Airbnb, Amazon, ...) as they make it easier to adjust rankings based on business needs.


The universal ranker will be implemented as a sequence of discovery 2.0 jobs, to be run within the feed service (more specifically within the homepage pipeline workflow). The diagram below shows the main steps, with the new universal ranker ones in blue:
  



Note that scoring runs twice in order to work around current decoration bottlenecks (addressed separately) and avoid the added latency of an extra decoration (estimated at an additional 25ms p95 latency for fetching popular items from post pagination payload, this would increase further if done pre-pagination). The first scoring pass is used to score stores and truncate the number of entities that need to be decorated. The second scoring pass scores carousels and requires  UX decorations to filter out the nested stores[3].
Implementation
Scoring
In this first stage, each candidate entity is scored by a model.
Methodology
We have two options to rank all entities and the entity constituents:


* Horizontal Ranking → these are the existing rankers that exist for store carousels and similar
   * We return a score for an entity where we ignore the sequence of the entity’s constituents
   * This allows us to score both the entity constituents and the entity independently
* Vertical Ranking → this is the universal ranker
   * We need some ranker (possibly the universal ranker itself) to first rank all constituents inside an entity
   * Then we do a second request to the ranker to rank all entities
   * We can optionally consider the sequence of the constituents
      * E.g., store position inside the carousel may be a usable feature
   * At the cost of an extra pass, we can use different rankers inside each entity. This allows us to also support manual (operator-specified) rankings 


For the initial proposal we will consider 2-pass ranking in order to better support surfaces like promotions or operator-defined carousels that may want to use different rankers.
Features
We have four core classes of features for any model[o][p][q][r][s][t][u][v]
* User embedding
   * consumer2vec or average item purchase embedding  
* Entity embedding
   * An aggregation of the entity’s constituent embeddings 
      * For stores this will be store2vec
      * For items this will be item2vec
* Entity type code
   * An integer denoting the type of the entity (carousel, collection, store, etc)
* Entity metadata
   * Entity name 
      * Carousel name
      * Store name
      * Collection title[w][x][y][z][aa]
   * …


Which requires us to sent the following keys in a request:
* user_id
* the set of store_ids in the entity
* entity_code
* entity metadata codes
Embedding aggregation
Every entity is defined as a collection of embeddings, so a carousel is defined as a sequence[ab][ac] or average of the constituent store embeddings. Both definitions have different tradeoffs


* Average the embeddings of the entity constituents 
   * This can generalize to entities with a large number of constituents but may have poor performance on entities with low-quality internal rankings
* Using a sequence (via concatenation or position coding) of embeddings
   * We are limited to some maximum sequence length for the entity which may be truncated
   * We will give a low rank to carousel that have inefficient rankings
   * This treatment requires a two-pass ranker


We propose using sequence information because we generally observe that customers do not scroll beyond the first ~3 results in a carousel so we expect the truncation problem to be minimal.


The overall model structure would look like below:
  



Where we use separate encoders for user embeddings and the constituents of each entity. The universal ranker would consume these constituent embeddings along with an embedding type code and the user embedding to predict the target variable. 


This architecture allows us to use embeddings which are computed on-the fly for content that may be dynamically adjusted on the home page (e.g., ads, promos or stores with ETA as a feature). For example, we may wish to have a specific deals/offers encoder which uses store and incentive information to embed an offer in real-time.


In the MVP we will use pre-trained embeddings and will not be building separate constituent encoders, [ad][ae][af][ag][ah]but in the future we will support this using Sibyl composite models. In particular, our framework does not need to distinguish between pre-trained embeddings and embeddings that are computed in real-time. An alternative proposed in the appendix is to use end-to-end training to learn constituent embeddings during training.

Training objective
The model is trained on the probability of conversion on the entire entity 
* Label: conversion event on any store inside the entity
   * For a store this is just a regular conversion event
   * For a carousel or collection any conversion on any store inside the collection will lead to a positive label
* Prediction: probability of conversion on that entity
Evaluation
We evaluate model performance on AUC on the entity conversion predictions and labels.


We will evaluate the model on two baselines:
* Against the current performance of the carousel ranker + home feed
* Against home feed only performance
Caveats
* This design does not provide a “discount” by vertical position which may bias results towards those at already high ranks
* Probability of entity conversion does not make any assumptions about the entities inside the carousel, and may have a very different probability estimate vs the one given by the current carousel ranker
* The model needs the carousel contents pre-selected, so this cannot ensure the optimal allocation of stores to carousels/available slots on the home feed
   * This kind of optimal matching would require an estimate of all <store, position> conversion probabilities

Debuggability/Interpretability
This model does not have any explicit goals around improving debuggability or interpretability. Due to the heavy dependence on pre-computed embeddings, the features are not very interpetable. In order to evaluate the model performance on recommendations we will perform:
1. Extensive manual testing via testing parties
2. A/B testing to meet the specified product goals above
Alternative designs
* Status-quo
   * We continue to hard-code entity type order or rely on new suboptimal heuristics to intermix content. The cost increases as new types are introduced (for example store items).
* Tree-based model scoring
   * This would have the advantage of having been previously implemented in Sibyl with relatively known latencies. However tree models face some major limitations right now due to limitations with embedding support in Sibyl (e.g., we can't do feature engineering on sets of embeddings). This could be a good thing to test in the future.
* Re-calibrating the scores of the carousel and LGBM ranker and training a single model from that
   * This architecture is sensitive to the scores from the horizontal ranker and does not scale to tiles or collections yet. If we have problems using the two-tower implementation this is a reasonable fallback (discussed in the risks and mitigations sections below).
Risks and Mitigation
1. The current proposed architecture cannot be productionized due to high latency/load. This model architecture is somewhat novel at DoorDash at this scale and consequently there is some risk feed service cannot handle the cost of the inference step
   1. Mitigations [Risk: Medium]
      1. We will build a dummy model with a two tower architecture to test latency changes.
      2. If the latency is too high, then we will test a simpler baseline using LGBM or LR models that create calibrated predictions for entities and store cells while working with the ML Platform team to improve neural network model serving performance
      3. There are more details in the risks section of the Eng RFC
2. The current architecture is not performant in offline testing compared to the current baseline.
   1. Mitigations [Risk: Low]
      1. We believe this is highly unlikely due to the extremely low performance of the current baseline (efficiency metric values ~0.5 which is effectively random). However if this is an issue, we will address this through further feature engineering (e.g, including the first ranking stage’s predictions)
3. We see large gains in offline evaluation metrics but these results do not translate into online gains on the home page.
   1. Mitigations [Risk: Low/Medium]
      1. Due to the v1.7.2 experiment we believe this is unlikely to be a major risk. However, this is a problem similar to what we saw in 2020Q4 and 2021Q1. Our hypothesis that this problem is due to inefficient rankings, so if ranking efficiency gains do not translate to homepage attributed conversion gains, we will need to further investigate the relationship between offline and online metrics.
      2. A systematic investigation here would entail tuning multiple rankers for different efficiency (or AUC) levels and trying to understand the impact of cannibalization between search and home page

Future work (out of scope for MVP)
* Model incentives explicitly when ranking entities with deals/offer badges...
   * Motivation: The MVP does not have explicit representation for incentives, likely leading to worse performance. One idea is to build embeddings for promotions. Ideally, we would have a unified representation which factors in incentives.
* Support more entities (banners, categories, etc.) on more pages (cross-vertical homepage).
   * Motivation: The MVP only ranks stores, carousels, and collections on the default home page. As a next step, we plan to extend this by adding more features and models to achieve better global conversion.
* Revisit deduplication logic
   * Motivation: The MVP reuses existing entity deduplication logic to limit the scope of ranking changes. However a previous experiment had shown that deduplication has a significant impact on conversion.
* Boost diversity.
   * Motivation: User experience and global conversion typically benefit from diversifying the results after scoring. Experimenting with an approach similar to Airbnb’s would be a good first step in that direction.
* Allow model objectives beyond pConv (VP, multi-objective, …). 
   * Motivation: This is helpful to better balance business needs (past attempt).
* Supporting vector based model output values for intermediate layers in composite models [JIRA]
   * Motivation: In order to rank content that dynamically[ax][ay][az][ba][bb] changes (e.g., free form text or ads/promos where the incentives may be adjusted in real-time) we need to be able to compute embeddings in realtime. Intermediate vector based model outputs allows us to load an encoder to compute an embedding in real-time which is then used as an input to the ranker.

Mark’s ideas on possible architectures to test (HD version)
  

Above is a simplified design based on deepFM method. This arch uses our existing pre-trained embeddings (store, dish and query) instead of end2end training. Since feed ranking doesn’t have a strong intent signal as search ranking has, we need to leverage as many user’s historical engagement signals as possible to learn their intent and then match it to relevant items (store or carousel). Therefore, there are two important sets of features for a feed ranking model:
1. user engagement data (e.g. stores / dishes user ordered / clicked / viewed / etc., queries user searched)
2. item (store / dishes / query) attributes such as vertical, cuisine, daypart, brand
By having those two sets of features, we can create many pairs of features. For example, given one item attribute feature (e.g. cuisine), we can create the corresponding user side feature, such as a list of cuisine types the user engaged in the past 30 days. All of those pairs of user side and item side features are fed to the DL model to learn the user's intent and find the most relevant item. Since we don’t have embeddings for item attributes (e.g. cuisine), we cannot create corresponding user side features. One solution could be creating two one-hot encoding features, one for the item and the other one for the user and treat them as dense features. As long as the cardinality of an item attribute is not very high (e.g. cuisine), then it’s worth testing.


This design above just gives us a taste of the type of arch and features that I think would work based on my experience. I’m still new (2nd week) so that I’ve no idea what features are available / logged and am not sure how big the model size that our ML platform can handle. If this arch works, we can further test the following more complex modules:
1. Carousel Position Weight: So far, we plan to concatenate the carousel and using a FC layer to process that. We could also have learnable weights to learn the carousel position importance when pool the embeddings. It might give us a better result.[be][bf]
2. Attention/Gating (e.g. DIN model): The user engaged items are highly “correlated” with the items to rank. For example, one user ordered Chinese food and fast food, then for an asian store / dish, the conversion rate prediction should rely more on the Chinese food order history. Therefore, adding attention to the items user engaged could improve the model performance and personalization.
3. AttentionFM: This is an extension on top of the Attention method above. This method assumes that for different feature interactions, its importance may vary for different item prediction.
4. resNet (wiki): This method can help to capture both lower and upper level representation. And if our predictions are off in certain segments (e.g. city, cursion type), this method could also fix that to a certain extent and improve the overall model performance.
5. End2end Training (depends on our ML platform). [bg][bh][bi][bj][bk][bl][bm][bn]This can enable us to use any item attribute (especially sparse attribute) available without pre-trained embeddings.[bo][bp][bq][br][bs]
