Blocker 1

PR review complexity

BlockerWhat happens
Reviewer bandwidthEngineers are bottlenecks for each other. Reviews sit for days.
InconsistencyDifferent reviewers catch different things. Friday PRs get rubber-stamped.
Knowledge fragmentationBest practices live in 15+ Confluence docs nobody reads mid-review.
The problem isn't lazy engineers. It's that thoroughness doesn't scale with human bandwidth. There are 10 things to check, 20 PRs in the queue, and one reviewer.

The Tools

Two agentic skills, built on Claude Code

/review-pr

Removes PR review complexity

  • 10 dimensions reviewed simultaneously
  • Searches Confluence for applicable rules
  • Self-verifies: reads source code, writes failing tests
  • Posts as PENDING review. Human submits.

/sandbox-test

Removes testing complexity

  • Engineer describes what to test, agent handles how
  • Drives real browser, captures 5 signal sources
  • 3 independent loads, cross-load comparison
  • Runs in background. Engineer is free.

Self-verification

It doesn't just flag things. It reads the source code to check if it's right. Then it writes a unit test to prove the bug exists. If the test passes, it drops the finding.

Reads our Confluence

Before reviewing, it searches Glean for relevant internal docs. It found "events >1MB silently dropped" and "Double.NaN caused a SEV-0" from pages I didn't even know existed.

Why This Works

Skills as encoded standards

  • Those Confluence docs nobody reads? They're checklists now. 15+ internal docs, checked on every PR automatically.
  • Friday 5pm = Monday 10am. The skill doesn't get tired or rush. Same rigor every time.
  • Corrections stick. Fix something once, it's fixed forever. The skill doesn't forget.
  • It's just markdown files. Want a new check? Edit one file. No deploy, no CI, no ticket.
  • Human still makes the call. The agent does the tedious scanning. You decide what's worth blocking on.
15
The Full Loop

Run it on your own code before opening the PR

Before you open

Run /review-pr on your own branch. Fix what it finds. Run /sandbox-test to collect evidence. By the time you open the PR, it's already been reviewed and tested.

When you review others

Point /review-pr at any PR you're added to. Read the findings in 5 minutes instead of reading the diff for 30. Focus your human attention on the judgment calls.

The result: you free up your time AND other engineers' time. Reviewers get a pre-validated PR with structured evidence. The human-in-the-loop step is reading findings and clicking "submit," not doing the grunt work.

The skills work the same whether you're the author or the reviewer. Same rigor, same evidence, same 10 dimensions.

Takeaways

Answering the three questions

What is a high quality PR? One that's been checked across 10 dimensions, not just the 2-3 the reviewer happened to think about. With structured testing evidence, not a screenshot.
What are the blockers? PR review complexity (bandwidth, inconsistency, tribal knowledge) and testing complexity (30-60 min manual validation, no structured evidence).
How to remove them? Encode your standards into agentic skills. They run every time, miss nothing, and each correction makes the next run better. This already found a real bug that would have shipped.
